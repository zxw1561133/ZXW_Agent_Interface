## 二、地图瓦片存放位置与数据格式

### 2.1 瓦片实际存放路径（磁盘）

- **路径**：`F:\jk\天津滨海\cursor\map_test\tiles`
- **配置位置**：`map/server.py` 中的 `TILES_DIR`
- **说明**：瓦片不在本工程目录内，而是单独放在上述路径；修改路径时需同时改 `map/server.py` 和前端提示用到的 `frontend/js/core/Config.js` 中的 `LOCAL_TILES_PATH`。

### 2.2 瓦片目录结构（z/x/y）— 按实际存放核对

瓦片按 **缩放级别 z / 列 x / 行 y** 组织，与常见瓦片规范一致。当前实际目录情况如下：

**实际存在的层级与格式：**

- **z（缩放级别）**：当前数据为 **1～10**（无 0 级）；z 越大比例尺越大、越放大。
- **x**：瓦片列号（整数，从 0 起）。
- **y**：瓦片行号（整数，从 0 起）。
- **文件格式**：实际存放均为 **`.jpg`**（未发现 `.png`）。

**目录树示例（与实际一致）：**

```
F:\jk\天津滨海\cursor\map_test\tiles\
├── 1\                    # 缩放级别 1
│   ├── 0\
│   │   ├── 0.jpg
│   │   └── 1.jpg
│   └── 1\
│       ├── 0.jpg
│       └── 1.jpg
├── 2\
│   └── ...               # 若干 x 子目录，其下为 y 子目录与 .jpg
├── 3\
├── 4\
├── 5\
├── 6\
├── 7\
├── 8\
├── 9\
└── 10\                   # 缩放级别 10，x 较多（如 0,1,…,100,1000,…）
    ├── 0\
    ├── 1\
    ├── …
    └── 1023\             # 示例：高 zoom 时 x 可达千级
        └── *.jpg
```

- **z**：缩放级别，当前为 1～10；数值越大越放大。
- **x / y**：瓦片列、行号；高 zoom 时 x、y 会很大（如 10 级下 x 有 0～1000+）。
- **文件格式**：实际均为 **`.jpg`**；若将来有 `.png`，前端 `buildTileUrl` 可传 `format='png'`。

### 2.3 瓦片访问 URL

- **服务地址**：`http://127.0.0.1:9001`（地图服务启动后）
- **单张瓦片 URL 规则**：`http://127.0.0.1:9001/{z}/{x}/{y}.jpg`（当前实际均为 `.jpg`；若日后有 `.png` 可传 `format='png'`）
- **示例**：`http://127.0.0.1:9001/1/0/0.jpg`、`http://127.0.0.1:9001/10/512/384.jpg`
- **前端拼接**：`frontend/js/core/ApiService.js` 中的 `buildTileUrl(z, x, y, format)`，依赖输入框中的「瓦片源」地址作为 base。请求时 z 建议在 1～10 范围内与现有数据一致。

## 三、网格数据存放位置与数据格式

### 3.1 网格数据文件位置

- **目录**：`F:\jk\天津滨海\cursor\earth-3d-platform\grid`（即工程根目录下的 `grid/`）
- **读取方**：`backend/server.py`，通过 `GRID_DIR` 指向该目录。

### 3.2 任务与文件对应关系

| 任务类型（API 参数） | 数据来源 |
|----------------------|--------|
| initGrid（初始网格） | Test_grid_task1.json 内的 `initGrid` 键（不读单独文件） |
| task1Grid（任务 1）  | Test_grid_task1.json |
| task2Grid（任务 2）  | Test_grid_task2.json |
| task3Grid（任务 3）  | Test_grid_task3.json |
| groups（分组）      | Test_group.json |

后端在 `backend/server.py` 的 `file_map` 中写死上述映射；**初始网格**从 task1 文件中的 `initGrid` 读取，不读 `Test_grid_init.json`。

### 3.3 网格 JSON 数据格式（按当前 grid 目录实际内容）

**单任务网格（initGrid / task1Grid / task2Grid / task3Grid）**

- 根节点：一个对象，键为**任务名**（`initGrid`、`task1Grid`、`task2Grid`、`task3Grid`），值为网格数组。
- **初始网格**：来自 `Test_grid_task1.json` 中的 `initGrid` 键（不单独存文件）。该数组内元素可含 `altitude`，与任务 1 同构。
- **任务 1**：与 initGrid 同文件，后端请求 task1Grid 时若文件仅有 `initGrid` 键会 remap 为 `task1Grid` 返回。
- **任务 2/3**（Test_grid_task2/3.json）：根键为 `task2Grid`/`task3Grid`，元素含 `altitude`，`length`/`width` 为浮点数（如 3.0）。

| 字段 | 类型 | 说明 |
|------|------|------|
| gridIndex | 整数 | 网格编号 |
| latitude | 浮点数 | 纬度 |
| longitude | 浮点数 | 经度 |
| length | 数值 | 长度（init 中为整数 3，task 中为 3.0） |
| width | 数值 | 宽度（同上） |
| altitude | 浮点数 | 高程（仅 task1/2/3 存在） |

示例 — 初始网格（来自 `Test_grid_task1.json` 的 `initGrid` 键）：

```json
{
  "initGrid": [
    {"gridIndex": 10, "latitude": 34.46, "longitude": 108.02, "length": 3.0, "width": 3.0, "altitude": 2420.0},
    {"gridIndex": 20, "latitude": 34.45, "longitude": 108.11, "length": 3.0, "width": 3.0, "altitude": 2420.0}
  ]
}
```

示例 — 任务网格（`Test_grid_task2.json` / `Test_grid_task3.json` 同构）：

```json
{
  "task2Grid": [
    {
      "gridIndex": 200,
      "latitude": 34.974179187785374,
      "longitude": 108.4868731353545,
      "length": 3.0,
      "width": 3.0,
      "altitude": 2300.0
    }
  ]
}
```

**分组数据（Test_group.json）**

- 根节点：`{"groups": [...]}`，`groups` 为数组，每项为一个分组对象。
- 每个分组对象包含：
  - **members**：对象，键为分组内序号（字符串 `"0"`,`"1"`,…），值为**平台 ID 字符串数组**（如 `["101"]`、`["201","209","217"]`），表示该组包含哪些平台。
  - **platformGridMaps**：数组，每个元素为一张「平台在各任务网格中的索引」映射表：
    - `platformID`：平台 ID（字符串，与 members 中引用一致）
    - `initGridIndex`、`task1GridIndex`、`task2GridIndex`、`task3GridIndex`：该平台在对应任务网格中的 gridIndex（或索引约定，依前端使用方式）。

示例（片段）：

```json
{
  "groups": [
    {
      "members": {
        "0": ["101"],
        "1": ["102"],
        "2": ["201", "209", "217"]
      },
      "platformGridMaps": [
        {
          "platformID": "201",
          "initGridIndex": 10,
          "task1GridIndex": 118,
          "task2GridIndex": 218,
          "task3GridIndex": 355
        }
      ]
    }
  ]
}
```

### 3.4 网格 API 接口

- **列表**：`GET http://127.0.0.1:9000/api/grid/list`  
  - 返回：`["initGrid", "task1Grid", "task2Grid", "task3Grid", "groups"]`
- **某任务数据**：`GET http://127.0.0.1:9000/api/grid/data?task=initGrid`（或 task1Grid、task2Grid、task3Grid、groups）  
  - 返回：对应 JSON 文件**完整内容**（根键与文件名约定见上）；若文件不存在，后端返回模拟数据。

## 四、服务与端口

| 服务       | 端口 | 说明 |
|------------|------|------|
| 后端 API   | 9000 | 提供前端静态资源 + `/api/health`、`/api/grid/list`、`/api/grid/data` |
| 地图瓦片   | 9001 | 提供瓦片图片，URL 形式为 `/{z}/{x}/{y}.jpg` 或 `.png` |

- 前端页面：浏览器访问 `http://127.0.0.1:9000`（由后端 9000 提供 `frontend/`）。
- 瓦片源配置：前端默认使用 `http://127.0.0.1:9001`，可在页面「瓦片源」输入框中修改。

## 五、关键配置汇总

| 配置项 | 位置 | 含义 |
|--------|------|------|
| 瓦片目录（服务端） | map/server.py → TILES_DIR | 瓦片在磁盘上的路径 |
| 瓦片地址（前端） | Config.js → SERVER.TILE_SERVER | 默认瓦片服务 base URL |
| 本地瓦片路径提示 | Config.js → LOCAL_TILES_PATH | 仅用于界面提示，与 TILES_DIR 一致 |
| 网格数据目录 | backend/server.py → GRID_DIR | 指向项目根下 grid/ |
| 任务→文件名映射 | backend/server.py → file_map | 见上文「任务与文件对应关系」 |

如需更换瓦片盘符或目录，需同时修改 `map/server.py` 的 `TILES_DIR` 和前端 `Config.js` 的 `LOCAL_TILES_PATH`（以及必要时检查输入框默认值）。
